# SPI Test Bench Documentation

## 1. Overview

This document explains the SPI module in the STM32 Test Bench project.

The SPI module verifies SPI communication using a loopback test. The STM32 sends one byte from the SPI MOSI pin and receives the same byte back on the SPI MISO pin.

The test is executed through Zephyr Shell over UART and automated using:

- Host Python script
- Robot Framework test suite

---

## 2. Module Scope

This SPI module belongs to the driver group:

```text
DIO
ADC
SPI
```

The SPI module is isolated from PWM and UART modules owned by other project members.

```text
STM32F401 Blackpill
```

### SPI Peripheral

```text
SPI1
```

### SPI Pins

| Signal | STM32 Pin | Description |
|---|---:|---|
| SPI1_SCK | PA5 | SPI clock generated by STM32 master |
| SPI1_MISO | PA6 | Master input, receives data |
| SPI1_MOSI | PA7 | Master output, transmits data |
| SPI1_NSS | PA4 | Not used in current loopback test |

---

## 4. SPI Loopback Wiring

The current test uses SPI loopback.

Only MOSI and MISO are connected together:

```text
PA7 / SPI1_MOSI  ----  PA6 / SPI1_MISO
```

SCK is not connected externally in this test.

```text
STM32F401 Blackpill

        SPI1
    +-----------+
PA5 | SCK       |  Not connected
PA7 | MOSI  ----+----------------+
PA6 | MISO  <---+----------------+
PA4 | NSS       |  Not used
    +-----------+

Loopback wire:
PA7 ---- PA6
```

### Why SCK is not connected

The STM32 is configured as SPI master. During `spi_transceive()`, the STM32 generates the SPI clock internally on PA5. Since the test loops MOSI back to MISO on the same board, no external slave device is used, so SCK does not need to be connected to another device.

## 10. SPI Source File

File:

```text
Zephyr_Firmware/src/testbench_spi.c
```

The SPI module contains:

```text
SPI device definition
SPI configuration
SPI initialization function
SPI byte transfer helper
Zephyr Shell commands
```

### SPI device

```c
#define SPI_NODE DT_NODELABEL(spi1)

static const struct device *spi_dev = DEVICE_DT_GET(SPI_NODE);
```

### SPI configuration

```c
#define SPI_FREQUENCY_HZ 1000000U
#define SPI_TEST_BYTE 0xA5

static struct spi_config spi_cfg = {
    .frequency = SPI_FREQUENCY_HZ,
    .operation = SPI_OP_MODE_MASTER |
                 SPI_TRANSFER_MSB |
                 SPI_WORD_SET(8) |
                 SPI_LINES_SINGLE,
    .slave = 0
};
```

| Field | Meaning |
|---|---|
| `SPI_OP_MODE_MASTER` | STM32 acts as SPI master |
| `SPI_TRANSFER_MSB` | Most significant bit is sent first |
| `SPI_WORD_SET(8)` | SPI frame size is 8 bits |
| `SPI_LINES_SINGLE` | Standard MOSI/MISO mode |
| `frequency` | SPI clock frequency is 1 MHz |

### SPI initialization

```c
int testBench_spi_Init(void)
{
    if (!device_is_ready(spi_dev)) {
        return -ENODEV;
    }

    return 0;
}
```

## 11. SPI Transfer Logic

The module transfers one byte using `spi_transceive()`.

```text
TX byte -> MOSI -> loopback wire -> MISO -> RX byte
```

ASCII flow:

```text
tx_byte
   |
   v
TX buffer
   |
   v
spi_transceive()
   |
   v
PA7 MOSI -------- PA6 MISO
   |
   v
RX buffer
   |
   v
rx_byte
```

If the loopback wire is correct, the received byte must equal the transmitted byte.

Example:

```text
TX 0xA5
RX 0xA5
```

## 12. Shell Commands

The SPI module registers two shell commands:

```text
tb_spi_test
tb_spi_transfer <byte>
```

### Command: `tb_spi_test`

This command sends a fixed byte:

```text
0xA5
```

Expected response:

```text
SPI_TEST PASS TX 0xA5 RX 0xA5
```

If the received byte is different:

```text
SPI_TEST FAIL TX 0xA5 RX 0x00
```

### Command: `tb_spi_transfer <byte>`

This command sends a user-selected byte.

Examples:

```text
tb_spi_transfer 0x55
tb_spi_transfer 0xA5
tb_spi_transfer 0xFF
tb_spi_transfer 129
```

Expected responses:

```text
SPI_TRANSFER TX 0x55 RX 0x55
SPI_TRANSFER TX 0xA5 RX 0xA5
SPI_TRANSFER TX 0xFF RX 0xFF
SPI_TRANSFER TX 0x81 RX 0x81
```

If no argument is passed:

```text
Usage: tb_spi_transfer <byte>
```

If the byte is out of range:

```text
Error: byte must be between 0x00 and 0xFF
```

## 15. Host Python Test

File:

```text
Host/spi_test.py
```

The Python script:

1. Opens `/dev/ttyUSB0`.
2. Sends `tb_ping`.
3. Sends `tb_spi_test`.
4. Sends multiple `tb_spi_transfer` commands.
5. Checks that TX and RX match.
6. Exits with status `0` on PASS or `1` on FAIL.

Run:

```bash
cd ~/zephyrproject/iti_applications/Test-Bench/Host
python3 spi_test.py
```

Expected final result:

```text
SPI Test: PASS
```

## 17. Robot SPI Validation Keywords

The Robot Python library can include SPI validation keywords in:

```text
Robot_Framework/resources/testbench_serial.py
```

Useful SPI keywords:

```text
SPI Test Response Should Be Valid
SPI Transfer Response Should Match
```

These keywords parse the shell response, extract TX and RX bytes, and check that they match.

Example Robot usage:

```robot
SPI Loopback Test
    ${response}=    Send Shell Command    tb_spi_test
    SPI Test Response Should Be Valid    ${response}

SPI Transfer 0x55 Test
    ${response}=    Send Shell Command    tb_spi_transfer 0x55
    SPI Transfer Response Should Match    ${response}    0x55
```

## 19. Troubleshooting

### Case 1: `RX 0x00`

Example:

```text
SPI_TEST FAIL TX 0xA5 RX 0x00
```

Possible causes:

| Cause | Fix |
|---|---|
| PA7 and PA6 are not connected | Connect PA7 to PA6 |
| Wrong pins used | Check board pinout |
| MISO is floating | Ensure loopback wire is connected |
| SPI pinctrl not applied | Check `build/zephyr/zephyr.dts` |
| Wrong SPI peripheral | Confirm SPI1 is enabled |

Debug command:

```bash
grep -R "spi1_sck_pa5\|spi1_miso_pa6\|spi1_mosi_pa7" build/zephyr/zephyr.dts
```

### Case 2: Command not found

Example:

```text
tb_spi_test: command not found
```

Possible causes:

| Cause | Fix |
|---|---|
| `testbench_spi.c` not added to CMake | Add it to `target_sources()` |
| Firmware not flashed | Run `west flash --runner openocd` |
| Board not reset after flash | Press reset button |

### Case 3: Serial port busy

Example:

```text
Could not open serial port /dev/ttyUSB0
```

Possible cause:

```text
picocom is still open
```

Fix:

```text
Ctrl + A
Ctrl + X
```

Or:

```bash
sudo fuser -k /dev/ttyUSB0
```

### Case 4: SPI build error

Check that SPI is enabled:

```conf
CONFIG_SPI=y
```

Check that overlay enables SPI1:

```dts
&spi1 {
    status = "okay";
    pinctrl-0 = <&spi1_sck_pa5 &spi1_miso_pa6 &spi1_mosi_pa7>;
    pinctrl-names = "default";
};
```
